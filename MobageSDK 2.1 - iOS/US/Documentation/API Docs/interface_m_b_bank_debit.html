Detailed Description

Interface for purchasing virtual in-app items. To complete a purchase, your app typically follows these steps:

  1. Call createTransaction:comment:withCallbackQueue:onSuccess:onCancel:onError: (MBBankDebit), which creates a transaction and presents a purchase screen to the user. If the user completes the purchase, the transaction's state changes from kMBTransactionStateNew to kMBTransactionStateAuthorized.
  2. Call openTransaction:withCallbackQueue:onSuccess:onError: (MBBankDebit), which changes the transaction's state to kMBTransactionStateOpen and places the user's funds into escrow.
  3. If the user's funds are successfully placed into escrow, deliver the virtual item to the user.
  4. Call closeTransaction:withCallbackQueue:onSuccess:onError: (MBBankDebit) to capture the user's funds and change the transaction's state to kMBTransactionStateClosed.

Important: Your app must wait for the successCB callback from each method before proceeding to the next step.

onSuccess: (MBTransaction *transaction)  successCB
onError: (MBError *error)  errorCB 

Mark the transaction as canceled, setting its state to kMBTransactionStateCanceled and releasing funds from escrow if applicable. Call this method if any of the following occur:

  • The user cancels the transaction.
  • The user does not have sufficient funds for the transaction.
  • The virtual item cannot be delivered.
  • An error occurs that prevents the transaction from being processed.
Parameters:
transactionIdThe transaction's unique ID.
cbQueueThe dispatch queue for the callback.
successCBCalled if the transaction is successfully canceled. The transaction parameter contains information about the transaction.
errorCBCalled if an error occurs while canceling the transaction. The error parameter contains information about the error.

+ (void) continueTransaction: (NSString *)  transactionId
withCallbackQueue: (dispatch_queue_t)  cbQueue
onSuccess: (MBTransaction *transaction)  successCB 

Continue processing a transaction created by an app server using the Mobage REST API. When a transaction is created through the REST API, its state is set to kMBTransactionStateNew. Calling this method on the transaction does the following:

  1. The transaction's state is set to kMBTransactionStateAuthorized.
  2. The user's funds are placed into escrow.
  3. The transaction's state is set to kMBTransactionStateOpen. The virtual item should now be delivered to the user.

If the user has insufficient funds for the transaction, the transaction's state is set to kMBTransactionStateCanceled, and the cancelCB callback is called.

Parameters:
transactionIdThe transaction's unique ID.
cbQueueThe dispatch queue for the callback.
successCBCalled if the user's funds are successfully placed into escrow. The transaction parameter contains information about the transaction.
cancelCBCalled if the user has insufficient funds for the transaction. The transactionId parameter contains the transaction's unique ID, and the reason parameter contains the reason why the transaction was not completed.
errorCBCalled if an error occurs while processing the transaction. The transactionId parameter contains the transaction's unique ID, and the error parameter contains information about the error.

+ (void) createTransaction: (MBBillingItem *)  billingItem
comment: (NSString *)  comment
withCallbackQueue: (dispatch_queue_t)  cbQueue
onSuccess: (MBTransaction *transaction)  successCB 

Create a new transaction. When your app calls this method, the following steps occur:

  1. The app's inventory is checked to make sure the requested item exists, and a new transaction is created, with its state set to kMBTransactionStateNew.
  2. Mobage presents a screen that prompts the user to confirm the transaction.
  3. If the user decides not to proceed with the transaction, the cancelCB callback is called, and the transaction is automatically canceled. Otherwise, the transaction's state is set to kMBTransactionStateAuthorized.
Parameters:
billingItemThe billing item for the transaction.
commentA comment on the transaction. On the US/worldwide platform, this comment is not currently displayed to users.
cbQueueThe dispatch queue for the callback.
successCBCalled if the transaction is successfully authorized. The transaction parameter contains information about the transaction.
cancelCBCalled if the user cancels the transaction, or if the user has insufficient funds for the transaction.
errorCBCalled if an error occurs while creating the transaction. The transactionId parameter contains the transaction's unique ID, or nil if no transaction was created. The error parameter contains information about the error.
